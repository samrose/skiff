# Skiff — Phase 1 Spec (consolidated)

This is the consolidated user brief for Phase 1 of the skiff project, as captured from the planning conversation on 2026-05-23. It is the input to `docs/superpowers/plans/2026-05-23-phase-1-npm-streaming.md`.

---

## Project name

`skiff`

## What we're building

A system that processes new npm package publishes and produces Nix-built, content-addressed, sandboxed package artifacts in a public binary cache. Phase 1 is the streaming-only slice — no batch reconciliation, no devenv integration. The goal is a working pipeline that takes npm publishes in and produces signed cache entries out.

## Stack

- **Language:** Go (single repo, single Go module, multiple binaries under `cmd/`).
- **Orchestration:** Temporal (Go SDK `go.temporal.io/sdk`).
- **Nix:** Determinate Nix. Assume `nix` on PATH is Determinate. Don't use Determinate-only builtins in published derivations — keep them portable across CNix/Lix/Determinate.
- **Object storage:** Garage S3 (S3-compatible). AWS SDK for Go v2 with configurable endpoint.
- **Metadata store:** ClickHouse, via `github.com/ClickHouse/clickhouse-go/v2`.
- **Binary cache:** Cachix-compatible HTTP cache (narinfo + nar.xz). Self-host backed by Garage.
- **Signing:** ed25519 in Nix binary-cache key format. Generate a key pair at setup; sign all published narinfos.

## Containerization (added 2026-05-23 mid-conversation)

All Docker images are **Linux-based**, including the three skiff binaries (ingest/worker/resolver). The full pipeline runs inside docker-compose Linux containers — the macOS dev host just runs `docker compose up`. The worker container has Determinate Nix installed with a persistent `/nix` volume; all Nix derivations target `x86_64-linux` or `aarch64-linux` matching the container architecture, never Darwin. `nix develop` on macOS is purely for editing and `go test`.

## Open source (added 2026-05-23 mid-conversation)

Skiff is intended to be open source and available for anyone to use.

Decisions taken during planning:
- **License:** MIT
- **Module path / GitHub URL:** `github.com/skiff-build/skiff`
- **Deployment posture:** self-host today, hosted public cache eventually (Phase 2+). README and docs frame for both.
- **Contribution gate:** DCO sign-off (`Signed-off-by:` on every commit, enforced by GitHub Action).
- **Code of Conduct:** Contributor Covenant 2.1.
- **Container image registry:** `ghcr.io/skiff-build/skiff-{ingest,worker,resolver}`, built and pushed by GitHub Actions.

## Decisions on open questions (resolved during planning)

| Question | Resolution |
|---|---|
| Garage vs MinIO for local dev | Garage (both dev and prod, one mental model) |
| ClickHouse schema | Defined in the plan: `events`, `package_observations`, `classifications`, `builds`, `cache_publications`, `ingest_state`, `schema_migrations` |
| Temporal task queue topology | Single shared queue `skiff-default`, one worker binary. Concurrency: 100 workflow tasks, 50 activities |
| Cache signing key handling | Both `CACHE_SIGNING_KEY` (raw) and `CACHE_SIGNING_KEY_FILE` (path) supported; file path wins when both set |

## Repo layout

```
/cmd
  /ingest       — registry change stream consumer
  /worker       — Temporal workflow + activity workers
  /resolver     — HTTP server exposing the cache lookup API
/pkg
  /registry     — npm registry client (changes feed, tarball fetch)
  /classify     — pure functions: tarball + metadata -> classification
  /build        — invokes `nix build`, manages sandboxes
  /cache        — writes narinfo/nar.xz, signs, uploads to Garage
  /store        — ClickHouse access for metadata
  /workflows    — Temporal workflow + activity definitions
/nix
  /packager.nix — the Nix expression that builds a single npm package
                  given (tarball path, name, version, classification)
  /flake.nix    — dev shell + the packager derivation
/deploy
  /docker-compose.yml — local dev: Temporal, ClickHouse, Garage, and skiff binaries
/docs
  /architecture.md — short description of the pipeline
  /classification.md — the classification rules and their rationale
```

## What Phase 1 must do

1. Subscribe to the npm registry's `_changes` feed (the CouchDB-style change stream at `https://replicate.npmjs.com/_changes`). Track the last-seen sequence number in ClickHouse so restarts resume cleanly. Rate-limit politely.

2. For each new `(name, version)` tuple observed, fire a Temporal workflow `ProcessPackage(name, version)` that:
   1. Fetches the tarball from the registry, verifies the integrity hash from the packument matches.
   2. Stores the source tarball in Garage under `sources/<sha512>/`.
   3. Runs the classifier on the unpacked tarball + package.json.
   4. Writes a classification record to ClickHouse.
   5. If `pure_js`, fires `BuildPackage` as a child workflow.
   6. If anything else (`has_lifecycle_script`, `has_native_code`, `fetches_at_install`, `broken`, `suspicious`), records the reason and stops. No build attempt in Phase 1.

3. `BuildPackage(name, version, nodejs_version, system)` activity:
   1. Generates a Nix expression invoking `nix/packager.nix` with the source tarball, name, version.
   2. Runs `nix build` in a sandboxed environment. No network access to the build sandbox beyond fetching the already-downloaded tarball.
   3. On success, signs the resulting store path and publishes it to the cache layout (narinfo + nar.xz) in Garage.
   4. Writes a build record to ClickHouse.

4. The resolver HTTP server exposes:
   - `GET /resolve/:name/:version` → JSON with cache URL if built, or structured "not cached" response with the classification reason.
   - `GET /cache/...` → standard Cachix-compatible cache endpoints.

5. Observability:
   - Structured JSON logs to stdout from all binaries.
   - Prometheus metrics endpoint on each binary (`/metrics`).
   - ClickHouse holds the source-of-truth event log.

## Classifier rules for Phase 1 (precedence order)

| Class | Rule |
|---|---|
| `broken` | tarball won't unpack, package.json missing/invalid |
| `suspicious` | install script contains `curl ... \| sh`, `wget ... \| sh`, base64-decoded execution patterns, references to `~/.ssh`, `~/.aws`, `/etc/passwd`, `/etc/shadow`, or env vars likely to contain secrets (`AWS_*`, `GITHUB_TOKEN`, `NPM_TOKEN`) |
| `has_native_code` | `binding.gyp` present, `*.node` files in tarball, `*.c`/`*.cc`/`*.cpp` files in tarball |
| `fetches_at_install` | install script contains `http://`, `https://`, references to `node-pre-gyp`, `prebuild-install`, `@mapbox/node-pre-gyp` |
| `has_lifecycle_script` | `preinstall`, `install`, or `postinstall` in `package.json` scripts (and not already classified above) |
| `pure_js` | none of the above |

Each rule is a pure function `classify(tarball, packageJson) → Classification` with a reason field. Unit tests with fixture tarballs.

## The packager.nix derivation

A function `{ tarball, name, version, nodejs }` → derivation that:
- Unpacks the tarball.
- Does NOT run any npm install scripts.
- Lays out the result as a `node_modules`-style entry: a directory ready to be symlinked into a consumer's `node_modules/<name>`.
- Output is a Nix store path.

Phase 1 builds `pure_js` packages only. Unpack-only behavior is the whole point: content-addressed, sandbox-built artifact that exactly mirrors the tarball's intended-installed contents minus any lifecycle scripts.

## Local development

`nix develop` in the repo root drops the dev into a shell with: Go toolchain, golangci-lint, Temporal CLI, ClickHouse client, Garage CLI / AWS CLI, Determinate Nix (matching the system Nix).

`docker compose -f deploy/docker-compose.yml up` brings up: Temporal (with SQLite), ClickHouse single-node, Garage single-node, **and** the three skiff binaries (ingest, worker, resolver) as Linux container services.

The binaries are configurable via env vars: `TEMPORAL_HOST`, `CLICKHOUSE_DSN`, `S3_ENDPOINT`, `S3_BUCKET`, `CACHE_SIGNING_KEY` / `CACHE_SIGNING_KEY_FILE`, and others (see plan for complete list).

## Explicitly out of scope for Phase 1

- No override repo, no per-package patches.
- No native code builds. Classification only.
- No fetches-at-install handling beyond classification.
- No multi-architecture builds. Target only the worker container's system.
- No multi-Node-version targets. Default Node 20.
- No devenv integration.
- No SLSA provenance for cached package artifacts (Phase 2). Note: container-image SLSA provenance is a free win from `docker/build-push-action` and is accepted.
- No batch reconciliation workflow. Streaming only.
- No public deployment surface in Phase 1 (Phase 2+ goal).

## Deliverable

When Phase 1 is done, the user should be able to:
1. Bring up the local stack with docker compose.
2. (Implicit: skiff binaries run inside the compose stack.)
3. Watch the changes feed populate ClickHouse with classifications.
4. See `pure_js` packages get built and published to the local cache.
5. Curl the resolver for a known `pure_js` package and get back a cache URL.
6. Add that cache URL as a substituter in a clean Nix install and pull down the built package as a store path.

## How to work

- Start with the repo skeleton, OSS scaffolding, and the local dev stack. Make `nix develop` and `docker compose up` work before writing business logic.
- Then implement the registry client and ingest path. Get changes flowing into ClickHouse before touching Temporal.
- Then add Temporal workflows. Start with `ProcessPackage` and the classifier.
- Then add `BuildPackage` and the cache publishing path.
- Then add the resolver.
- Then observability + e2e smoke.
- Then CI, image publishing, and the v0.1.0 release.

Write tests as you go. The classifier especially needs good fixture-based tests.
